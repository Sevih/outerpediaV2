### Data Access Layer (`src/lib/data/`)

**NEVER import JSON files directly in components.** Always use the data access layer.

```typescript
// FORBIDDEN
import chars from '@data/game/characters/alice.json'

// REQUIRED
import { getCharacter } from '@/lib/data/characters'
const alice = await getCharacter('alice')
```

Every data access function lives in `src/lib/data/`. This abstraction allows:
- Changing file structure without touching components
- Adding cache, validation, lazy loading
- Proper TypeScript typing on all returns

### Pipeline (`pipeline/`)

Single unified pipeline replacing scattered scripts.

```bash
npm run pipeline              # Run all steps
npm run pipeline:step chars   # Run one step
npm run pipeline:validate     # Validation only
```

All file paths are centralized in `pipeline/config.ts`. When moving a file, update ONE line.

## Internationalization (i18n)

### Supported Languages
`en` (English), `jp` (Japanese), `kr` (Korean), `zh` (Chinese)

### Language Type
```typescript
type Lang = 'en' | 'jp' | 'kr' | 'zh'
```
Always use the `Lang` type from `@/types/common`. Never hardcode language arrays or union types elsewhere.

### Localization Patterns

**Game data** (from parser): suffix fields вЂ” `name`, `name_jp`, `name_kr`, `name_zh`
```typescript
type WithLocalizedFields<T, K extends string> = T & {
  [P in `${K}_${'jp' | 'kr' | 'zh'}`]?: string
}
```

**Editorial content** (human-written): LangMap objects
```typescript
type LangMap = Partial<Record<Lang, string>>
// { en: "Hello", jp: "гЃ“г‚“гЃ«гЃЎгЃЇ" }
```

### Localization Functions
- `l(obj, field, lang)` вЂ” Get localized suffix field with English fallback
- `lRec(record, lang)` вЂ” Get value from LangMap with English fallback

### Routing
- **Dev**: `localhost:3001/jp/characters/alice` (path segment `[lang]`)
- **Prod**: `jp.outerpedia.com/characters/alice` (subdomain via middleware)
- Code always uses `[lang]` App Router segment. Middleware handles subdomain rewriting.

### Inline Tags
Format `{X/value}` for inline component injection in text. These tags stay **identical across all languages**.

- `{B/EffectName}` вЂ” Buff
- `{D/EffectName}` вЂ” Debuff
- `{E/Element}` вЂ” Element with icon
- `{C/ClassName}` вЂ” Class
- `{P/CharacterName}` вЂ” Character link
- `{S/StatName}` вЂ” Stat (ATK, SPD, etc.)
- `{EE/Name}` вЂ” Exclusive Equipment
- `{SK/CharName|SkillType}` вЂ” Skill reference

## Next.js Conventions

- **Server components by default**, `'use client'` only when needed (hooks, interactivity)
- `useMemo` for filters/sorts depending on props or state
- Images: avoid fixed `width`/`height` вЂ” use container with `position: relative` + `<Image fill sizes="Xpx" className="object-contain" />`

## Coding Standards

- No emojis unless explicitly requested
- Character names must match `data/game/characters/*.json` filenames exactly
- Slugs are kebab-case, used as primary identifiers (never filter/group on localized fields)
- Stats in requirements are strings, not numbers
- Proactively flag optimization opportunities (architecture, security, performance)
